结论

CANN Bench 调用的是 submission 暴露的 Python 函数。它不直接调用 Triton 编译 API，也不需要理解 kernel 语法。

核心决策
把 Triton-Ascend 作为现有 CANN submission 合同的一种实现方式。 框架执行 outputs = func(**updated_params)；用户函数执行 _exp_kernel[grid](...)，后者才进入 Triton JIT、缓存查询和 NPU launch。

端到端运行链路

wheel 构建和 kernel JIT 是两个不同阶段。前者仅打包 Python；后者在首次 launch 当前 specialization 时发生。

01 / BUILD源码打包

build.sh 生成纯 Python cann_bench wheel，不做 NPU AOT 编译。

02 / INSTALL强制安装

--force-reinstall --no-deps 替换同版本旧 submission，不改平台依赖。

03 / DISCOVER接口发现

重新 import 包，扫描公开 callable，按 proto.yaml 匹配函数。

04 / LAUNCHJIT 与执行

Python wrapper 计算 grid；kernel[grid](...) 查询缓存并按需编译。

05 / EVAL验证与计时

同步 NPU，比较 Golden；pre-flight 后进入 CANN profiler。

JIT 到底在哪里

@triton.jit 在 import 时把 Python 函数包装成 JITFunction，并不立即生成 binary。真正的执行边界是带 grid 的调用。

submission 触发代码
grid = (triton.cdiv(n_elements, _BLOCK_SIZE),)
_exp_kernel[grid](
    x, output, n_elements,
    scale, shift,
    math.log(base) if base > 0 else 0.0,
    HAS_BASE=base > 0,
    BLOCK_SIZE=_BLOCK_SIZE,
)
Triton-Ascend 的 [] 入口
def __getitem__(self, grid):
    return lambda *args, **kwargs: self.run(
        grid=grid,
        warmup=False,
        *args,
        **kwargs,
    )

_exp_kernel[grid] 返回记录 grid 的闭包；紧随其后的参数调用才进入 JITFunction.run()。

01cann_bench.expwrapper 分配输出、准备 grid 和编译期属性。
02KernelInterface.__getitem__构造绑定 grid 的 launcher closure。
03JITFunction.run绑定参数，得到签名、specialization 和 constexpr，查询进程内 cache。
04triton.compiler.compile查询磁盘 cache；未命中才运行 Ascend lowering pipeline。
05kernel.run载入 npubin，并把 compiled kernel 提交到当前 NPU stream。

  1. 包名固定为 cann_bench。

实际运行的 Exp 核心源码

下面内容与 wheel 中的 cann_bench/exp.py 一致。wrapper 只做布局、输出和 grid；公式、exp 与 store 全部位于 kernel。

cann_bench/exp.py 验证 wheel：32838d2e…ce14 
"""Triton-Ascend implementation of the CANN Bench Exp interface."""

from __future__ import annotations

import math

import torch
import triton
import triton.language as tl


_BLOCK_SIZE = 4096


@triton.jit
def _exp_kernel(
    x_ptr,
    output_ptr,
    n_elements,
    scale,
    shift,
    log_base,
    HAS_BASE: tl.constexpr,
    BLOCK_SIZE: tl.constexpr,
):
    offsets = tl.program_id(0) * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)
    mask = offsets < n_elements
    value = tl.load(x_ptr + offsets, mask=mask, other=0.0).to(tl.float32)
    value = value * scale + shift
    if HAS_BASE:
        value = value * log_base
    tl.store(output_ptr + offsets, tl.exp(value), mask=mask)


def exp(
    x: torch.Tensor,
    base: float = -1.0,
    scale: float = 1.0,
    shift: float = 0.0,
) -> torch.Tensor:
    """Return ``exp((x * scale + shift) * log(base))`` on an NPU tensor.

    ``base <= 0`` selects the natural-base form used by the task schema. The
    wrapper only prepares metadata and output storage; the numerical work runs
    in ``_exp_kernel``.
    """
    if not x.is_contiguous():
        x = x.contiguous()

    output = torch.empty_like(x)
    n_elements = x.numel()
    if n_elements == 0:
        return output

    grid = (triton.cdiv(n_elements, _BLOCK_SIZE),)
    _exp_kernel[grid](
        x,
        output,
        n_elements,
        scale,
        shift,
        math.log(base) if base > 0 else 0.0,
        HAS_BASE=base > 0,
        BLOCK_SIZE=_BLOCK_SIZE,
    )
    return output

怎么知道用户包里有什么

wheel 本质是 ZIP。安装前看到的是静态内容，import 后看到的是公开接口，运行时 Guard 看到的是部分行为；这三层不能互相替代。

层次能回答的问题当前实现
wheel 静态内容文件、包名、METADATA/RECORD、是否携带二进制本次已人工核验；自动强制审计尚未实现
import 后接口公开 callable 和函数签名scan_interfaces() 已实现
运行行为违规 PyTorch API、NPU→CPU 规避TorchOpGuard 与 DeviceResidencyGuard 已实现

生产环境需补的 import 前审计
  • 路径穿越、symlink 与文件类型白名单
  • 默认拒绝 submission 自带可执行文件和未知 `.so`
  • 校验 RECORD，并记录提交整体 SHA256
  • Python AST/import 策略检查
  • 非 root、无网络、资源受限容器执行
当前安全边界 import cann_bench 会执行用户顶层 Python 代码。因此“import 后扫描”不能证明包没有副作用；--no-deps 和运行时 Guard 也不能替代 import 前审计与容器隔离。

是否可以修改 Triton-Ascend 优化

现象优先修改层判断依据
单个算子的 grid、block、访存或地址计算问题submission局部代码即可解释和复现，不应扩大 compiler 影响面
多个算子共同出现相同 lowering、vector math 或冗余 castTriton-Ascend backend保存 TTIR/ttadapter/npubin，以共同 IR 证据证明
接口发现、同步或 profiler 不能覆盖 Triton launchCANN Bench只有框架级可复现失败才新增专用逻辑
已验证案例
SwiGLU 先做 submission 级优化，不改 compiler。 将逐元素除法/取模寻址改为二维 grid，case 1 从 642.72 μs 降至 21.16 μs，约改善 30.4 倍。只有 Exp、Sigmoid、Mish 等多个数学算子显示相同 lowering 瓶颈时，才进入 backend。
